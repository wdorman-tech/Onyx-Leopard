# Building a production-grade LLM business simulation on CAMEL-AI OASIS

**No open-source platform yet combines LLM-powered agents with rigorous economic modeling for full-spectrum business simulation — this is a genuine greenfield opportunity.** MiroFish, the closest existing project, uses OASIS purely for social-media dynamics prediction and lacks financial models, supply chains, or market mechanisms entirely. The recommended architecture layers deterministic mathematical models (handling ~80% of agent decisions at near-zero cost) beneath an LLM reasoning layer (for strategic and creative decisions), orchestrated through a hybrid simulation engine that blends Agent-Based Modeling, System Dynamics, and Discrete Event Simulation — the same multi-method approach that makes AnyLogic the industry standard, but open-source and LLM-native. CAMEL-AI OASIS provides the scalable multi-agent infrastructure (tested to 1 million agents) and Claude API integration, but its social-media-centric platform, action types, and state management must be substantially replaced with economic primitives. The existing Black Jaguar stack (FastAPI + Next.js 15 + React Flow + Recharts) is architecturally sound and requires only targeted additions — Apache ECharts for real-time streaming, DuckDB for analytical queries, and LanceDB for semantic caching that can reduce LLM costs by ~47%.

## The mathematical foundation requires three simulation layers

The platform's three-tier scope — single company, competitive market, full ecosystem — maps cleanly onto established mathematical frameworks that should be implemented as composable modules.

**Layer 1 (Single Company + Supply Chain)** rests on three model families. System Dynamics, formalized by Forrester (1961) and codified in Sterman's *Business Dynamics* (2000), models inventory, cash flow, workforce, and production capacity as stocks connected by flow rates with feedback delays. The canonical Beer Distribution Game demonstrates how even a four-tier supply chain (retailer → wholesaler → distributor → factory) generates the bullwhip effect through information delays and bounded rationality. The stock-and-flow equation **dS/dt = Inflow(t) − Outflow(t)** governs all accumulation processes. Discrete Event Simulation (DES), implemented via SimPy, handles stochastic operational processes — order processing, production lines, warehouse operations, shipping — where queueing theory (M/M/c models, Little's Law **L = λW**) provides analytical foundations. Operations Research models supply the decision logic: **EOQ (Q* = √(2DS/H))** for reorder quantities, the Newsvendor critical fractile **F⁻¹(Cᵤ/(Cᵤ + Cₒ))** for inventory under uncertainty, and linear/mixed-integer programming (via PuLP or Google OR-Tools) for production scheduling and resource allocation.

**Layer 2 (Competitive Market Dynamics)** draws on game theory and econometrics. The Cournot model (quantity competition with Nash equilibrium **q* = (a−c)/[b(n+1)]**), Bertrand model (price competition yielding the paradox of marginal-cost pricing with just two firms), and Stackelberg leader-follower model provide the strategic interaction framework. Price elasticity models (**ε = (∂Q/∂P)(P/Q)**, typically estimated via log-linear regression) drive demand response to pricing decisions. For demand forecasting, ARIMA models handle individual product series while Vector Autoregression captures cross-product and macroeconomic interactions. Stochastic processes generate realistic market dynamics: **Geometric Brownian Motion (dS/S = μdt + σdW)** for baseline price evolution, Merton jump-diffusion **(dS/S = (μ−λk̄)dt + σdW + JdN)** for sudden shocks, and Hamilton's Markov regime-switching model for business cycle transitions between expansion and contraction regimes.

**Layer 3 (Full Market Ecosystem)** requires ABM as the orchestration framework, with Leontief Input-Output analysis **(x = (I−A)⁻¹d)** mapping inter-firm flows, and network theory (betweenness centrality, clustering coefficients, scale-free properties) identifying supply chain vulnerabilities. The Inoperability Input-Output Model (Santos & Haimes, 2004) directly models how disruptions cascade through interdependent firms. Mesa 3 (Python) serves as the recommended ABM engine — it provides agent scheduling, network spaces, data collection, and browser visualization, with a new experimental DiscreteEventSimulator that bridges ABM and DES. **Each agent internally invokes the appropriate mathematical model** — OR for operational decisions, game theory for competitive strategy, econometrics for forecasting — while the LLM layer handles qualitative reasoning, negotiation, and adaptive strategy.

Key foundational references include Tesfatsion's *Handbook of Computational Economics, Vol. 2* (2006) for ACE methodology, Miller & Blair's *Input-Output Analysis* (2009) for I-O models, and the open-source implementations: PySD for system dynamics, statsmodels for econometric models, NetworkX for graph analysis, and pymrio for multi-regional I-O analysis.

## What OASIS actually provides and what must be built from scratch

OASIS (arXiv:2411.11581, Apache 2.0, 2.4k stars) is a five-pillar system: Environment Server, Agent Module, Recommendation System, Time Engine, and Simulation Engine. It uses a PettingZoo-style interface (`env.reset()` → `env.step(actions)` → `env.close()`) with discrete time-stepping at **3 simulated minutes per tick**. Agents communicate through async channels, state persists in SQLite, and LLM backends route through CAMEL's ModelFactory supporting OpenAI, Anthropic (full Claude support including Opus/Sonnet/Haiku), and VLLM for local models. The system has been validated at scales up to one million agents with load balancing across multiple LLM instances.

**What OASIS gives you that is directly reusable:** scalable concurrent agent infrastructure with async execution, CAMEL framework integration providing native Claude API support (streaming, function calling, extended thinking), PettingZoo-compatible step-based simulation interface, agent profile management and social graph infrastructure, SQLite-based state logging, and the Time Engine for discrete simulation steps. The `SocialAction` class wraps agent communication through channels — this pattern extends to business actions.

**What must be built from scratch is substantial.** OASIS has **23 social-media action types** (CREATE_POST, LIKE_POST, FOLLOW, MUTE, etc.) that must be replaced entirely with business actions: PLACE_ORDER, SET_PRICE, PRODUCE, SHIP, NEGOTIATE, HIRE, INVEST, REPORT_EARNINGS. The Environment Server simulates Twitter/Reddit feeds and must be replaced with market order books, supply chain state, and financial ledgers. The Recommendation System (interest-based content discovery) must become a market discovery and price signaling system. Most critically, **OASIS has no numerical state management** — agent state is LLM-text-driven, not equation-driven. Business simulation demands precise financial tracking (balance sheets, inventory counts, prices to the cent), market clearing mechanisms, and deterministic mathematical model execution that OASIS does not support.

The Electronic Mall feature (added December 2024 on OASIS's Reddit platform) represents the closest OASIS gets to economic simulation and provides the extensibility pattern. TwinMarket (NeurIPS 2025, ICLR 2025 Best Paper) demonstrates what LLM-based market simulation looks like — it uses BDI (Belief-Desire-Intention) cognitive architecture to reproduce fat-tailed returns, volatility clustering, and market bubbles with up to 1,000 agents — but it is an independent project, not built on OASIS.

Compared to alternatives, **Mesa** is better for equation-driven ABM with mathematical models (agents implement any formula in their `step()` method) but lacks LLM integration and scales to thousands, not millions. **Concordia** (Google DeepMind, Apache 2.0) offers perhaps the most elegant architecture for generative ABM — its Game Master pattern, modular component system, and marketplace component are directly relevant — but targets ~100s of agents. **AgentPy** provides clean Jupyter integration and parameter sensitivity analysis but has a smaller community. AnyLogic's multi-method approach (ABM + System Dynamics + DES in one model) is the conceptual gold standard but is proprietary Java software.

The recommended strategy: **use OASIS's scalable agent infrastructure and CAMEL integration as the foundation, but build new Platform types, Action types, and state management layers** rather than trying to adapt the social media components. Draw architectural inspiration from Concordia's modular component system and AnyLogic's multi-method hybrid modeling.

## How MiroFish works and why a business simulator is fundamentally different

MiroFish (AGPL-3.0, 18,000+ stars, backed by $4.1M from Shanda Group) is a social prediction engine built by BaiFu at BUPT in 10 days of "vibe coding." Its architecture is a three-tier system: **Vue 3 + Vite + D3.js** frontend, **Flask** backend with blueprints, and OASIS as the simulation engine with Zep Cloud (or Neo4j in forks) for GraphRAG-based knowledge graphs.

MiroFish's five-stage workflow — Graph Building (entity extraction via GraphRAG), Environment Setup (persona generation), Simulation (dual Twitter/Reddit platform via OASIS), Report Generation (ReACT-pattern ReportAgent with InsightForge and PanoramaSearch tools), and Deep Interaction (chat with any simulated agent) — is elegant for social prediction. The "God's-eye view" variable injection enables scenario branching mid-simulation. However, **MiroFish produces plausible narrative scenarios, not quantitative predictions** — it has no published benchmarks comparing predictions against actual outcomes, no supply/demand curves, no financial calculations, and no market clearing mechanisms.

The gap between social simulation and business simulation is fundamental. A business simulator needs **precise numerical state** (a company's cash balance is $2,347,891.23, not "the company has moderate financial resources"), **deterministic economic relationships** (price × quantity = revenue, not LLM-generated estimates), **market clearing** (matching buy and sell orders with actual transactions), and **temporal financial accounting** (income statements, balance sheets, cash flow statements that must balance). MiroFish's architecture is an excellent reference design for the LLM orchestration and reporting layers, but the simulation engine beneath must be fundamentally different.

Other critical prior art includes **EconAgent** (ACL 2024, Tsinghua) which demonstrated LLM agents reproducing the Phillips Curve and Okun's Law in macroeconomic simulation but only models household labor/consumption decisions, **InvAgent** (2024) which applies LLM chain-of-thought reasoning to multi-echelon supply chain inventory management via AutoGen, **LLM Economist** (2025, MIT License) which embeds persona-conditioned agents in two-tier Stackelberg games for tax policy with 3–1,000+ agents, and **Salesforce's AI Economist** (published in *Science Advances*) which uses two-level deep RL rather than LLMs for tax optimization in a grid economy. The Tsinghua survey repository (github.com/tsinghua-fib-lab/LLM-Agent-Based-Modeling-and-Simulation) catalogues the full landscape. **No existing project combines LLM-powered business agents + financial modeling + supply chains + market competition + organizational behavior in a single platform** — this is the greenfield opportunity.

## The hybrid decision engine is the critical architectural innovation

Running 100 LLM-powered agents for 100 simulation ticks at ~500 tokens per call generates **5 million tokens per simulation run**, costing $15–50+ at Claude Sonnet pricing. The solution is a hybrid architecture where deterministic mathematical models handle routine decisions and the LLM handles only strategic reasoning — research from Frontiers in AI (2025) found that hybrid colonies mixing LLM and rule-based agents often **outperformed purely LLM-based groups**.

The decision routing framework should classify every agent decision into four tiers:

- **Routine operations (~60% of decisions, zero LLM cost):** Inventory reorder via EOQ formula, production scheduling via LP, logistics routing via OR-Tools, financial accounting updates. These are pure mathematical computations.
- **Known heuristics (~20%, zero LLM cost):** Price adjustments within configurable bounds, standard supplier selection based on scoring algorithms, demand forecasting via ARIMA/state-space models. Rule-based with parameterized behavior.
- **Simple classification (~10%, Claude Haiku):** Categorizing market signals, sentiment analysis of simulated news events, routine competitive responses. Low-cost LLM for judgment calls.
- **Strategic/creative decisions (~10%, Claude Sonnet):** Market entry decisions, crisis response strategies, negotiation between firms, R&D investment allocation, responses to unprecedented scenarios. This is where LLM intelligence provides genuine value.

**Semantic caching** via LanceDB (embedded vector database, serverless like SQLite) can reduce even the LLM tier costs by ~47%, based on research on agentic plan caching (arXiv). Before calling Claude, search for semantically similar past queries in the vector store; if similarity exceeds 0.95, return the cached response. **Anthropic's built-in prompt caching** further reduces costs by reusing system prompts across calls. **Model tiering** (Haiku for simple, Sonnet for complex, Ollama with Qwen 7B for offline fallback) and **batch prompting** (grouping 10 similar agents into a single structured API call) complete the cost optimization strategy.

The simulation engine itself should use a **hybrid tick-based + event-driven loop**. Each tick: process queued events → gather agent decisions (LLM + deterministic in parallel via `asyncio.gather`) → apply decisions to state → record events to event store → broadcast to frontend via WebSocket → checkpoint periodically. Variable time scales handle the tension between hourly operational decisions and quarterly strategic ones: a tick represents one hour for operations, with strategic decision events triggered every 720 ticks (~1 month) and board-level LLM calls every 2,160 ticks (~1 quarter).

## The existing tech stack needs targeted additions, not replacement

**FastAPI** is confirmed excellent for this use case — native WebSocket support via Starlette, SSE via `StreamingResponse`, and async architecture that enables concurrent LLM calls through `asyncio.Semaphore` for rate limiting. A `ConnectionManager` class tracks active WebSocket connections and broadcasts simulation tick data. For background execution of long simulations, Celery with Redis provides job queuing and pub/sub for streaming results.

**Next.js 15** serves well as the framework shell, though Server Components and the App Router are largely irrelevant once the simulation is running — the real-time UI will be 95% client components. **React Flow** handles supply chain network visualization effectively up to **200–500 nodes** with proper optimization (wrap custom nodes in `React.memo()`, use Zustand for state management, enable `onlyRenderVisibleElements`); beyond that, Sigma.js with Graphology provides WebGL-based rendering for 100K+ edges. **Recharts should be supplemented with Apache ECharts** (`echarts-for-react`) for real-time streaming dashboards — ECharts is Canvas-based, handles 10M+ data points, and has built-in WebSocket streaming support, while Recharts (SVG-based) struggles above 5 updates/second.

The persistence layer should be a **dual-database architecture**. SQLite in WAL mode (`aiosqlite` for async) handles transactional simulation state — agent configurations, checkpoints, user settings — with fast point queries. DuckDB handles analytical queries on simulation history — columnar storage delivers 10–100x faster aggregations, GROUP BY, and window functions over time-series data. **LanceDB** (embedded, Rust-based, ~15ms for 5M vectors) provides vector storage for agent memory embeddings and semantic caching, with built-in table versioning that maps to scenario branching.

**Event sourcing** is the recommended state management pattern. Every agent decision, market update, and state transition is recorded as an immutable event in SQLite. Simulation state at any point can be reconstructed by replaying events from the nearest snapshot. Scenario branching works like Git: take a snapshot at any tick, fork the event stream with a new `simulation_id` and `parent_branch` reference, compare branches by querying events from the fork point. Snapshots every 50–100 ticks enable fast random access for time-travel debugging.

The complete recommended package set:

| Layer | Core | Additions |
|-------|------|-----------|
| **Backend** | FastAPI, uvicorn, camel-ai, anthropic | aiosqlite, duckdb, lancedb, celery, redis, litellm, msgpack, pydantic v2 |
| **Frontend** | Next.js 15, @xyflow/react, recharts | echarts-for-react, zustand, @react-sigma/core (future), framer-motion |
| **Simulation** | OASIS/CAMEL infrastructure | Mesa 3 (ABM engine), SimPy (DES), PySD (system dynamics), scipy/pulp (OR), statsmodels (econometrics), networkx (graphs) |

## The feature architecture should center on conversational simulation control

The chat interface uses **Claude's function calling / tool use** as a simulation controller. The LLM analyzes natural-language input, extracts intent, and generates structured JSON specifying function name and arguments — `create_company`, `inject_scenario`, `query_simulation_state`, `fork_scenario`, `advance_simulation` — which the backend executes against the simulation engine. A multi-step agent loop continues until the LLM produces a final text response, enabling complex multi-turn interactions like "Create a mid-size electronics manufacturer in Shenzhen with three tier-1 suppliers, then simulate what happens if the largest supplier goes bankrupt during peak season."

The UI should use a **split-pane layout**: chat panel (30–40% width) on the left with inline visualization cards (sparklines embedded in messages, action confirmation cards with predicted impact, context-aware suggestion chips), and simulation visualization (60–70% width) on the right. This right panel hosts four switchable views:

- **Network view** (React Flow): Hierarchical left-to-right layout from raw materials through tier 3/2/1 suppliers to manufacturer to distribution to retail. Custom node types for each entity class with embedded KPI badges. Animated edges with flowing dots showing material movement, thickness proportional to flow volume, color-coded green/yellow/red for health status.
- **Dashboard view** (Apache ECharts): Z-pattern layout with KPI cards across the top (revenue, costs, margin with sparklines and trend arrows), time-series charts in the middle, and detail tables at the bottom. Maximum 9 widgets per screen, grouped by Financial / Operations / Market clusters.
- **Sankey view** (Recharts `<Sankey>` or @nivo/sankey): Three flow types — material flow (raw → intermediate → finished), money flow (revenue → COGS → OpEx → profit), and information flow (orders → fulfillment → confirmation → payment).
- **Scenario tree** (React Flow as DAG): Horizontal branch visualization where each node is a simulation state snapshot. Branches diverge at event injection points. Side-by-side comparison with variance columns (absolute and percentage deltas) following Pigment/Causal UX patterns.

The **event injection system** supports four trigger types: user-triggered (via chat or event palette), scheduled (timeline markers on simulation progress bar), probabilistic (probability sliders per event type), and conditional (rule builder: IF inventory < X THEN trigger). Events cascade through the supply chain network using breadth-first propagation with configurable attenuation per hop and maximum hop depth. A preview overlay shows affected nodes before confirmation. Visual effects (pulsing red nodes propagating through the graph) make cascades viscerally understandable.

## Concrete implementation roadmap for a solo developer

**Phase 1 (Weeks 1–4): Foundation.** Implement the hybrid simulation engine with Mesa 3 as the ABM orchestrator, integrating EOQ, Newsvendor, and basic Cournot models as deterministic decision modules. Build the event sourcing layer in SQLite with snapshot/replay. Connect to Claude via CAMEL's ModelFactory with the four-tier decision router and semantic caching via LanceDB. Target: a single company with 5 suppliers, 1 factory, 3 distribution points running at <$0.50 per simulation.

**Phase 2 (Weeks 5–8): Visualization.** Build the React Flow supply chain editor with custom node types and animated edges. Add Apache ECharts dashboards for real-time KPIs. Implement the chat interface with Claude function calling for simulation control. Target: conversational company creation and visual monitoring of a running simulation.

**Phase 3 (Weeks 9–12): Competition and ecosystem.** Add game-theoretic models (Cournot/Bertrand) for multi-firm competition. Implement Leontief I-O for inter-firm flow structure. Build the scenario branching UI with comparison views. Add the event injection system with cascading propagation. Target: 3–5 competing firms in a market with supply chain disruption scenarios.

**Phase 4 (Ongoing): Scale and polish.** Add regime-switching models for business cycles, stochastic demand generation, geographic visualization for logistics, and parameter sensitivity analysis. Optimize for 50+ firms. Publish as open-source with documentation following CoMSES Net's ODD protocol for model transparency.

The key papers to implement against, in priority order: Sterman (2000) for system dynamics foundations, Park et al. (2023) for generative agent architecture, Horton (2023) for validating LLM agents as "Homo Silicus" economic actors, the OASIS paper (Yang et al., 2024) for scalable infrastructure patterns, and Santos & Haimes (2004) for supply chain disruption cascading via the Inoperability I-O Model. The key repositories: camel-ai/oasis for infrastructure, 666ghj/MiroFish for GraphRAG + reporting patterns, google-deepmind/concordia for modular GABM architecture, tsinghua-fib-lab/ACL24-EconAgent for LLM economic agent design, and zefang-liu/InvAgent for supply chain LLM agents.
